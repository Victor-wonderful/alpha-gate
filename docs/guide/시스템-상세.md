# Alpha Gate — 시스템 상세

> 이 시스템이 **내부에서 어떻게 동작하는지** 정리한 기술 문서. 실사용법은 [사용자 가이드](사용자-가이드.md) 참고.

---

## 1. 한 줄 정의

암호화폐 선물 트레이더가 진입 전에 **"이 거래를 해도 되는가"**를 검증하게 만드는 의사결정 도구. AI 분석(데이터 기반) + 거래 평가(등급) + 가상 거래 + 복기로 구성.

## 3. AI 분석 파이프라인 (핵심 도메인)

3단계 + 1단계 정리. 각 단계가 명확히 분리되어 있습니다.

### Stage 1 — Market Data (코드, 결정론적)
`buildSnapshot()` — Binance Futures에서 멀티 TF 캔들, 호가창, aggTrades, 펀딩비, OI, 24h 티커를 병렬 fetch. CoinGecko BTC 도미넌스. 자체 계산: 스윙 포인트, FVG, Order Block, Liquidity Zone, Volume Profile(POC/VAH/VAL), Order Flow. 스타일에 따라 다른 TF를 가져옵니다.

### Stage 2 — Strategy Agent (LLM, 가벼움)
`classifyStrategy()` — 스냅샷을 보고 **5개 전략 중 1개** 선택: `trend_pullback` / `breakout` / `range_fade` / `reversal` / `wait`. 출력 `{ primary, direction, confidence, reasoning, rejected }`. 평범한 한국어 강제, 시스템 프롬프트 캐싱.

### Stage 3 — Scenario Generator (LLM, 메인)
`synthesizeAnalysis()` — Strategy 결과를 입력으로 받아 시나리오를 그 전략 안으로 제약. 출력 `{ summary, structure, keyLevels, flow, scenarios[], actionNow, warnings }`. **스타일별 표준 손절/목표 범위를 강제**. max_tokens 3000.

### Stage 4 — Persist (코드)
`saveAnalysis()` — 분석 완료 시 Supabase `analyses` 테이블에 자동 저장. `?load=<id>`로 복원.

## 4. 트레이딩 표준 (수수료 차감 후 현실판)

왕복 비용 가정 **0.075%**(테이커+메이커 합산 최대). 절대 손절 하한 ≈ 0.225%.

| 스타일 | 손절폭 | 목표폭 | 최소 R:R | 기준 TF |
|--------|--------|--------|---------|---------|
| 스캘핑 | 0.3~1.2% (MTF ATR 1.5~2배) | 0.4~2% | 1.3+ | 1H/15M/5M |
| 데이 | 0.7~1.5% | 1.5~3% | 1.5+ | 4H/1H/15M |
| 스윙 | 2~5% | 5~15% | 2+ | 1D/4H/1H |
| 포지션 | 5~15% | 15~50% | 3+ | 1D/4H/1H |

> 스캘핑 밴드는 6코인 ~2년 A/B 백테스트로 검증(순R +0.067). 검증 하니스: `scripts/measure-bands.mjs`, `scripts/ab-bands.mjs`.

## 5. 등급 시스템

점수 합계 → 등급: ≥8 **A**, 5~7 **B**, 2~4 **C**, ≤1 **D**.

**가점:** 손익비 ≥3 +3 / ≥2 +2 / ≥1.5 +1, 손절 기준 명확 +2, BTC 정렬 +1, 박스 회피 +1, 거래량 동반 +1, 심리 체크 통과 +2.
**감점:** 손익비 무효 −2, 손절 과도 −1, 목표 비현실 −1, BTC 충돌 −2, 박스 중간 진입 −2, 뉴스 직후 −1, 연속 손실 −2, 심리 미충족 −1.

D등급은 원인을 구분: **계좌발**(일일 한도/과노출)="오늘은 보류", **셋업발**="고위험·10% 축소(막지 않음)". 하드 차단 대신 컨펌 플로우.

## 6. 포지션 사이징 (리스크 % 기반)

```
maxLoss     = accountSize × allowedLossPct
riskPerUnit = |entry − stop|
quantity    = floor(maxLoss / riskPerUnit × 10000) / 10000
positionSize = quantity × entry
```

용어: **잃을 한도**=거래당 리스크, **노출 금액**=positionSize, **매수 수량**=quantity, **필요 마진**=positionSize/leverage, **실효 손익비**=수수료 차감 후 R. 레버리지는 리스크와 무관, 필요 마진·청산 거리만 바꿉니다.

## 7. 주문 유형 & 예약 자동 판정

내부 `orderType`: `market` / `limit` / `stop`.
- **market**: 즉시 체결.
- **limit**(되돌림 대기): 롱=현재가가 내려와 지정가 도달, 숏=올라와 도달.
- **stop**(돌파 추격): 롱=현재가가 위로 트리거 돌파, 숏=아래로 이탈.

UI에서는 **"지금 바로 / 예약 주문"** 2택만 노출. 예약 시 limit/stop은 **예약가 vs 현재가 + 방향**으로 실시간 자동 판정:
- 롱: 예약가 ≤ 현재가 → limit, 초과 → stop
- 숏: 예약가 ≥ 현재가 → limit, 미만 → stop
- 현재가/예약가를 못 구하면 AI `orderHint`(없으면 limit) 폴백

## 8. 후보 레이더 (Candidate Radar)

`radar.ts` — 거래대금 상위 30개 크립토 무기한을 스타일별 4개 TF로 스캔. 신호: 유동성 sweep, 변동성 수축, 매물대(VAH/VAL), 펀딩 극단, 거래량 급증, 24h 극단. **긴-TF 핸디캡**으로 best-style 선정. 10분 cron(`scan-radar`)으로 `radar_candidates` 테이블에 적재.

**고정 자산:** `PINNED_SYMBOLS = [BTCUSDT, ETHUSDT, XRPUSDT, BNBUSDT]` — 점수 컷오프와 무관하게 항상 후보에 포함, 항상 진입 가능, 맨 앞 정렬. 상수는 클라이언트 안전을 위해 `radar-constants.ts`로 분리(server-only 누출 방지).

### 예상 진입 방향 — 데이터 검증으로 결정
방향은 **추세만** 사용: up→롱 / down→숏 / range→양방향.

> **근거:** 실거래 분석 291건(`analyses.strategy_direction`)을 ground truth로 신호별 방향 예측력을 측정. 추세 방향 **98.4% 적중**(기준선=무조건 숏 77.7% 대비 +20.7). sweep(49.6%)·펀딩(42%)·매물대(32%)·24h극단(43%) 등은 모두 기준선 이하(무작위~역효과)라 방향 판정에서 제외. 임의 가중치 휴리스틱은 69.7%로 기준선보다 나빠 폐기. **검증 하니스: `scripts/validate-radar-bias.mjs`, `scripts/validate-radar-rules.mjs`.** 새 방향 신호 추가 시 반드시 이 하니스로 검증할 것.

## 9. 자동 체결 / 정산 (Cron)

### 체결 — `limit-order-filler.ts` (`fill-limit-orders` cron)
`pending_limit_orders`(status='open')를 순회. 체결 조건(limit/stop × 방향)을 만족하면 마진 lock + `trades.order_status='filled'` + `pending_limit_orders.status='filled'`. 만료 시 `expired`, 잔액 부족 시 `canceled`. 만료 임박 텔레그램 경고.

### 정산 — `resolve-trades` cron
**진입한 거래(`order_status='filled'`)만** 대상으로 created_at 이후 캔들을 스캔해 손절/목표 도달을 판정, `result_r`·`exit_reason`·`closed_at` 기록 + 가상 지갑 정산. 만료 도달 시 시장가 자동 청산.

> ⚠️ **알려진 버그(수정됨, 2026-06-14):** 과거 resolve-trades가 `order_status` 필터 없이 `closed_at IS null`인 거래를 전부 가져와, **미체결 예약 주문(pending)을 트리거가로 진입했다 가정**하고 손절 정산해버렸다. 결과: 거래일지엔 손절로 기록되는데 가상거래엔 미진입으로 남는 불일치. → `order_status='filled'` 필터 추가로 해결. 동일 패턴(미체결을 진입으로 오인)을 다른 곳에 넣지 말 것.

## 10. DB 스키마 핵심

- **`trades`** — 진입 입력 + 진입 시 평가(pre_grade/score/rr) + 결과(exit_price/result_r/exit_reason/closed_at) + AI 복기 + `order_status`('pending'/'filled'/'canceled'/'expired') + `mode`('live'/'backtest') + `is_paper`. RLS 본인 행만.
- **`pending_limit_orders`** — 예약 주문(limit/stop). `status`('open'/'filled'/'canceled'/'expired'), `order_kind`, `limit_price`, `expires_at`.
- **`analyses`** — 분석 영구 저장. symbol/style/primary_strategy/strategy_direction/scenarios_count + snapshot/strategy/report JSONB.
- **`radar_candidates`** — 레이더 스캔 결과(trend/signals/best_style/style_atr 등).
- **`scenario_outcomes`** — 분석 시나리오별 가격/상태 추적 + watch(가격 도달 알림).
- **`notification_channels` / `notification_log`** — 알림 채널·발송 기록.
- **`paper_wallets`** — vUSDT 잔액·마진.

## 11. 알림 시스템

`notify.ts`(Telegram/Discord 어댑터) + `notify-dispatch.ts`(이벤트→채널). 이벤트: D등급 경고, 연속 손실, AI 복기 완료, 일일 요약, 분석 추천 시각, 시나리오 가격 도달, 만료 경고/자동 청산. 텔레그램은 딥링크(`/start <code>`)로 chat_id 자동 등록.

---

## 12. 백테스트 시스템

분석을 **과거 시점**으로 돌려(`buildSnapshot({ at })`, lookahead 방지) walk-forward 시뮬로 봉별 진입·손절/목표를 체결. 결과를 `mode='backtest'` 거래로 저장. 저널/성과에서 별도 탭으로 필터.

---

## 13. 주요 설계 결정 (역사)

1. 상단 탑바 → 좌측 고정 사이드바
2. AI 분석: 차트 이미지 단독 X → **데이터 기반 + 이미지 보조**
3. 단일 LLM → **3 stage 분리**
4. 결과: 심플 우선 + "전문가 정보 보기" 펼치기
5. 언어: 전문 용어 직접 사용 → **평범한 한국어 강제**
6. UI: 바이올렛 다크 테마 + 접이식 사이드바 (Pencil 시안)
7. 방향 추정: 임의 가중치 → **데이터 검증으로 추세 기반 확정**
